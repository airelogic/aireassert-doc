    Getting Started with AireAssert

    Writing an AireAssert test

    Jul 25, 2023

    Where do i start?

    You start here! This guide will walk you through the creation of your first AireAssert test of an AireForm. You will learn:

    • The structure of an AireAssert Test Set;
    • How to write a basic AireForm test;
    • How to run and debug the test locally.

    What we will be testing

    For this article we will be testing an example form which will show how you can get access to a repository and start creating tests!

    Getting Access

    Visit the AireAssert site and login using your email, if you do not see any repositories allocated to you on the left hand panel under “Repositories” then you should then contact your administrator who will be able to assign you a repository or assign you to a user group.

    Cloning the testset repository

    Clone the testset repo from bitbucket

    Creating your testset

    In the cloned repo, create a folder for your testset with a descriptive name, and in this place a Configuration.json file, in the following example it includes the URl where you targeted form exists and whether this form is formatted as a SinglePass or ClinicianPreview form.

    {
  "RuntimeUrl": "https://demo.forms4health.com",
  "TargetOrg": "",
  "UriFormat": "ClinicianPreview"
}

    Alongside the config file, create a folder called Features. This is where your tests will live.

    Writing your first feature (test)

    A feature file is a collection of test commands, which should not only run against a form, but also be easily readable. This means that not only are you testing, you are also documenting your form’s behaviour!

    While the files are written using Gherkin syntax you should be aware that it is a gherkin subset - you can only use the commands made available to you for the application you are testing, these can be found in the Command Library.

    Using any text editor, create a new file bmi-reporting.feature (this name is used as an example, this should be the name of your form to make it easier to differentiate from other forms) in the Features folder, and give it the following contents:

    Feature: BMI Reporting
   The BMI Reporting form allows patients to submit their height and
   weight information instead of having to visit a doctors surgery
   
Scenario: Underweight Patient
    Given we are completing the bmi-reporting eform
    When we set 'Height (m)' to '2'
    And we set 'Weight (kg)' to '60'
    Then the 'BMI' form section is visible
    And 'BMI' shows '15'
    And 'Weight appropriate' shows 'Underweight'

    Lets deconstruct what we have just written:

    • The feature title line is just that, a title;
    • We can add a block of descriptive text to help readers understand what behaviour this file is defining. This is optional but highly recommended;
    • A scenario. A feature can contain as many of these as you like.
    • Scenarios always start with a GIVEN line which loads the required form. The form name is the same as you will see in the URI when loading the form yourself;
    • A series of Whens and Thens. A When is an interaction with the form (doing something), while a Then is a check that the form has reacted in an expected way.

    Because the file is readable, you should be able to go to the form that you are basing your test off and walk through the scenario as if its running through the commands within the test!

    While describing every command is out of scope for this article there are a couple of things to note:

    • You refer to fields based on their labels. If your label contains single quotes (‘) just exclude these from the file (e.g. refer to a field “My item’s” as “My items”);
    • You can refer to a field by its ID if you prefer;
    • Field labels are case sensitive

    Pushing your changes

    Push your changes to the testsets repo back to git so that they can be picked up by the application.

    Running your test

    Please refer to the “Running a TestSet from the UI” section
